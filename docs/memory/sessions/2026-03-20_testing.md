# 20-03-2026 (Testing – Phase 2 Tier 1)

## Goal
Implement **Phase 2 Tier 1** of the testing strategy: extract revenue-critical business logic from API route handlers into pure, testable modules and wire them back. No new tools; same Vitest + Husky pre-push + GitHub Actions. Goal: protect the booking/payment/commission paths with fast unit tests so refactors don't break money flows.

## Done

### New modules (all in `apps/widget/src/lib/`)

1. **`reservation-rules.ts`**  
   Extracted from `apps/widget/src/app/api/reservations/route.ts`:
   - `validateRequiredFields(body)` — hotel vs direct channel; rejects missing experienceId, participants, totalCents, guestName, guestEmail.
   - `validateParticipants(participants, maxParticipants, isRental)` — 1..max for non-rental; skips for rental.
   - `validateRentalRequest({ requestDate, rentalDays, minDays, maxDays })` — start date required, rental days in range.
   - `validatePriceMatch(totalCents, expectedTotal)` — allows 1 cent rounding; rejects larger mismatch.
   - `computeRentalEndDate(startDate, rentalDays)` — inclusive end date (e.g. 3 days from Apr 1 → Apr 3); handles month/year/leap.
   - `normalizeRequestTime(time)` — `HH:MM` → `HH:MM:00`.
   - `resolveAutoDistributionRates(isSelfOwned)` — 92/0/8 (self-owned) vs 80/12/8 (default) for auto-created distributions (receptionist flow).

2. **`booking-rules.ts`**  
   Extracted from `apps/widget/src/app/api/webhooks/stripe/route.ts` (createBookingFromPayment):
   - `resolveCommissionRates({ isDirect, distribution })` — direct → SELF_OWNED; hotel → distribution row; errors if hotel and no distribution.
   - `resolveExperienceDate({ isRental, rentalStartDate, sessionDate, requestedDate })` — canonical date for emails/display.
   - `computeRentalDaysFromDates(startDate, endDate)` — inclusive duration (end − start + 1); undefined if either date missing.
   - `buildBookingRecord({ reservationId, sessionId, totalCents, split, paymentIntentId, chargeId })` — pure object for DB insert (amounts from split, status `confirmed`, paid_at now).
   - `verifyCommissionSplitIntegrity(split)` — asserts supplier + hotel + platform === total (used in tests with real `calculateCommissionSplit`).

3. **`cron-rules.ts`**  
   Extracted from cron routes (`auto-complete`, `expire-reservations`, `completion-check`):
   - `shouldAutoComplete(experienceDate, sevenDaysAgoCutoff)` — experience date ≤ cutoff (7+ days ago).
   - `shouldExpirePending(status, responseDeadline, now)` — pending and deadline < now.
   - `shouldExpireUnpaid(status, paymentDeadline, now, hasExistingBooking)` — approved, deadline passed, no booking (race protection).
   - `isCompletionCheckDue(experienceDate, yesterday)` — exact date match for "yesterday".
   - `resolveBookingExperienceDate(sessionDate, requestedDate)` — session date preferred over requested.

### Test files (Vitest, same pattern as Phase 1)

- **`reservation-rules.test.ts`** — 37 tests: required fields (hotel/direct), participants (range, rental skip), rental validation (min/max, null maxDays), price match (exact, ±1 cent), rental end date (1 day, 3 days, month/year/leap), time normalization, auto-distribution rates (92/0/8 vs 80/12/8).
- **`booking-rules.test.ts`** — 27 tests: commission resolution (direct vs hotel, missing distribution), experience date resolution (rental/session/requested), rental days from dates (same day, 3 days, nulls), buildBookingRecord (amounts, session_id null), verifyCommissionSplitIntegrity (real splits, 1 cent, large total).
- **`cron-rules.test.ts`** — 25 tests: shouldAutoComplete (7+ days, boundary, null), shouldExpirePending (deadline before/after, non-pending), shouldExpireUnpaid (with/without existing booking), isCompletionCheckDue (yesterday vs other days), resolveBookingExperienceDate.

### Route wiring

- **`reservations/route.ts`** — Uses `validateRequiredFields`, `validateParticipants`, `validateRentalRequest`, `validatePriceMatch`, `computeRentalEndDate`, `normalizeRequestTime`, `resolveAutoDistributionRates`. Removed inline validation and `DEFAULT_COMMISSION` import.
- **`webhooks/stripe/route.ts`** — Uses `resolveCommissionRates`, `resolveExperienceDate`, `computeRentalDaysFromDates`, `buildBookingRecord`. Removed direct `SELF_OWNED_COMMISSION` usage; booking insert is `buildBookingRecord(...)`.
- **`cron/auto-complete/route.ts`** — Uses `shouldAutoComplete`, `resolveBookingExperienceDate` for experience date and eligibility.
- **`cron/completion-check/route.ts`** — Uses `isCompletionCheckDue`, `resolveBookingExperienceDate`.

`expire-reservations` was not refactored: filtering is done in Supabase (pending + `response_deadline < now`, approved + `payment_deadline < now`), and the "skip if booking exists" check is per-row in code; the pure rules (`shouldExpirePending`, `shouldExpireUnpaid`) are tested for when you *would* expire, and could be used later if we move to in-memory filtering.

### Test and CI

- **Total:** 262 tests (was 239; +23 from Phase 1 count is actually +89 new tests across 3 new files — 262 is the full suite).
- **Run:** `pnpm test:pull` (from repo root) runs all app/package tests; widget alone: `pnpm --filter @traverum/widget test`.
- **Gate:** Husky pre-push runs `pnpm test:pull`; GitHub Actions runs same on push/PR to `main`.
- All 262 tests pass; runtime ~1–2 s.

## Decisions
- Extract pure decision/validation/date helpers into `*-rules.ts` modules next to existing `lib` code; routes stay thin orchestrators (load data → call rule → act).
- Keep idempotency and "booking already exists" inside the webhook handler (DB read); no extraction for that.
- Cron eligibility: extract only the date/eligibility predicates; keep Supabase queries in the route. `expire-reservations` still uses DB filters; rules document and test the *conditions* for expiry.
- Commission: `resolveCommissionRates` returns `{ rates, error }`; handler checks `rates` and logs `error`; `calculateCommissionSplit` (existing) does the math; `verifyCommissionSplitIntegrity` in tests guards the invariant.

# Phase 3: Playwright E2E Tests

## Goal
End-to-end browser and API integration tests using Playwright, running against a staging Supabase project (`aefmaxslyjlhtlulvgaa`). Validates golden path booking flows and API auth guards.

## Done

### Workspace (`apps/e2e/`)

- **`package.json`** — `@traverum/e2e` workspace, depends on `@playwright/test`
- **`playwright.config.ts`** — two projects (`ui` with Chromium, `api` headless); webServer starts widget dev server on port 3000; env loaded from `apps/e2e/.env`
- **`constants.ts`** — seed UUIDs and test values (hotel slug, experience ID, session ID, prices, guest data pattern)

### UI tests (mocked network for mutations, real SSR)

1. **`tests/ui/session-booking.spec.ts`** — Guest browses hotel, clicks experience, navigates to checkout, fills form, submits, redirected to (mocked) Stripe payment page.
2. **`tests/ui/request-booking.spec.ts`** — Guest browses hotel, clicks experience, navigates to request checkout, fills form, submits, redirected to reservation page with pending status.

### API tests (real staging Supabase)

3. **`tests/api/create-reservation.spec.ts`** — POST valid request-based payload: 200 + `reservationId`; verify row in DB (`pending`, correct `total_cents`). POST with missing `experienceId`: 400.
4. **`tests/api/dashboard-auth.spec.ts`** — POST accept with no auth header: 401. POST with garbage bearer token: 401.

### Helpers

- **`helpers/mock-api.ts`** — `mockSessionBookingAPI(page)` and `mockRequestBookingAPI(page)`: intercepts `POST /api/reservations` at browser level to return mock responses; intercepts Stripe redirect for session booking.
- **`helpers/widget-pages.ts`** — `fillCheckoutForm(page, data)` and `submitCheckoutForm(page)`: form interaction helpers.
- **`helpers/api-client.ts`** — `cleanupTestReservations()` (deletes `ghost-*@e2e.veyond.eu` rows) and `getReservation(id)` for DB assertions. Uses raw fetch against PostgREST with service role key.

### CI

- **`.github/workflows/e2e.yml`** — Triggers on PRs to `main` (path-filtered: `apps/widget/`, `apps/dashboard/`, `packages/shared/`, `apps/e2e/`). Caches Playwright browsers. Uploads report + traces on failure. Env vars from GitHub Secrets (`E2E_SUPABASE_URL`, `E2E_SUPABASE_ANON_KEY`, `E2E_SUPABASE_SERVICE_ROLE_KEY`, `E2E_STRIPE_SECRET_KEY`).

### Staging fixes

- **`apps/widget/src/lib/hotels.ts`** — Added fallback for missing `sort_order` column on distributions (retries with `created_at` ordering if PostgREST returns error code `42703`).
- **`apps/widget/next.config.js`** — Added `picsum.photos` to image remote patterns (seed data uses picsum for test images).
- **`20260321100000_sync_fk_names_and_missing_columns.sql`** — Extended to add all columns missing from staging: `guest_phone`, `is_request`, `requested_date/time`, `payment_deadline`, `stripe_payment_link_*`, `rental_*` on reservations; `sort_order` on distributions; FK renames; bookings columns; `hotel_id` nullable on reservations.

## How to run

```bash
# All E2E tests (starts dev server automatically if not running)
pnpm test:e2e

# UI tests only
pnpm test:e2e -- --project=ui

# API tests only
pnpm test:e2e -- --project=api

# With Playwright UI
pnpm --filter @traverum/e2e test:ui
```

Requires `apps/e2e/.env` with staging Supabase keys and Stripe test key.

## How to maintain

- **Seed data changes:** Update `apps/dashboard/supabase/seed.sql` and `apps/e2e/constants.ts` together.
- **New golden paths:** Add to `tests/ui/` (mocked) or `tests/api/` (real DB). Use `ghost-*@e2e.veyond.eu` emails for cleanup.
- **CI secrets:** Set `E2E_SUPABASE_URL`, `E2E_SUPABASE_ANON_KEY`, `E2E_SUPABASE_SERVICE_ROLE_KEY`, `E2E_STRIPE_SECRET_KEY` in GitHub repo settings.

## Decisions
- UI tests mock only mutations (POST /api/reservations), not SSR data. Pages render against real staging Supabase to catch schema/query issues.
- API tests use `ghost-*@e2e.veyond.eu` email pattern with `beforeAll`/`afterAll` cleanup to prevent test data accumulation.
- Dashboard auth tests are negative-only (401 checks). Positive accept flow would require a real supplier JWT + Stripe, so it's deferred.
- `sort_order` fallback in `hotels.ts` keeps production behavior while allowing staging DBs that haven't been fully synced.
